## Current Alpha Diagnosis

HW02 defined the real project goal clearly: this game is about map editing, visible A* reroutes and logistics changes, and interpretable AI decisions layered on top of that simulation. The previous HW04 draft did not stay focused on that goal. It described wrapper UI progress, but it did not honestly describe the actual alpha failures.

The current alpha had three verified blockers before this rewrite:

1. The hosted build link listed in the old HW04 report, `http://152.53.133.32:152/`, was not reachable on March 10, 2026. A dead link is not an alpha deliverable.
2. In the Active phase, the game-state overlay was still intercepting pointer events. That meant the player could not reliably drag the map, click tiles, or inspect entities, which directly broke the main interaction loop.
3. Workers could be pinned into AI or fallback-driven action targets such as `Harvest`, so they kept accumulating carry without reliably switching back to `Deliver`. Resources and objective progress could stall even when the simulation looked busy.

After re-checking the build and codebase, I found a fourth structural problem that is just as important:

4. The default scenario and objective chain did not force the player to edit the map in a meaningful way. Too much infrastructure already existed, so the simulation could drift forward without the player engaging the core mechanic from HW02.

This submission fixes the first-pass alpha blockers and rewrites the production plan around the actual game:

- Active-state overlay hit testing is now disabled during play, so the canvas receives input again.
- Camera interaction is clarified and restored: left click builds/selects, `Alt + Left Click` inspects, and right drag pans.
- Worker delivery is protected from fallback and policy steering, and sticky group targets such as worker `harvest` and trader `trade` are sanitized before runtime.
- Survival states such as `seek_food` are no longer allowed to override non-hungry workers through group policy.
- The default scenario is now sparse and build-first: one warehouse, two farms, one lumber site, minimal roads, and objectives that require expansion instead of passive waiting.
- The first objective is now a broken-frontier logistics repair: reconnect the west lumber route, reclaim the east depot with a warehouse, then scale to `4 farms`, `3 lumbers`, and `20 roads`.
- Weather fronts, bandit raids, trade caravans, and animal migration now bind to frontier zones instead of acting only as abstract numeric pressure.
- The HUD, Inspector, and Entity Focus panels now expose objective hints, frontier status, tile context, and entity decision summaries so stalls and reroutes are visible during play.

## System-by-System Upgrade Plan

### 1. Input / Camera

**Current weakness**

- Before the fix, the overlay still owned pointer hit-testing after `Start Run`.
- The interaction contract was unclear because right-click inspection conflicted with map drag.

**Why this misses the HW02 goal**

- If the player cannot directly drag, inspect, and edit the map, then the main gameplay loop does not exist.
- Any AI or routing system becomes irrelevant if the player cannot change the world.

**Upgrade plan**

- Keep the Active overlay fully non-interactive unless the run is in Menu or End state.
- Preserve the current interaction mapping: left click build/select, `Alt + Left Click` inspect, right drag pan.
- Add one regression gate for every submission: after `Start Run`, the viewport center must hit the `canvas`, and right drag must move the camera target.

### 2. Economy & Delivery Loop

**Current weakness**

- The original alpha allowed fallback or policy targets to keep workers in local action loops.
- Delivery and other survival-critical local states were not protected strongly enough.
- Group policy could still manufacture false hunger pressure by pushing `seek_food` onto workers who were not actually hungry.

**Why this misses the HW02 goal**

- If logistics do not close, then map edits do not have visible systemic consequences.
- A simulation that cannot reliably move resources from production to storage cannot support meaningful objective design.

**Upgrade plan**

- Keep local state priority strict: `eat`, `deliver`, and feasibility constraints beat group steering.
- Continue sanitizing group targets so they cannot issue sticky local action states such as `harvest`, `trade`, `eat`, `seek_food`, or `idle` for workers and traders.
- Add regression coverage for three cases:
  - workers with carry must eventually enter `deliver`
  - workers with no hunger pressure must not be pulled into `seek_food`
  - resources must increase or decrease in a way that changes objective progress

### 3. Map Editing & Navigation

**Current weakness**

- Earlier builds let the player place structures, but the consequences were inconsistent or too soft.
- The scenario often started with enough roads and buildings that reroute pressure was weak.

**Why this misses the HW02 goal**

- HW02 was not about decorating a map. It was about changing navigation and then observing reroute and redistribution behavior.

**Upgrade plan**

- Keep the sparse alpha layout so the colony begins as a partial network, not a solved network.
- Keep the west route gap and east depot reclaim step in the opening scenario so road continuity and storage placement remain mandatory.
- Tune default objectives so the player must add roads, storage, and production in order to unlock progression.
- Push more of the difficulty into path structure: warehouses, road continuity, chokepoints, and walls should produce visible A* changes.

### 4. Objectives & Scenario Design

**Current weakness**

- The previous objective chain leaned too hard on abstract colony progression and not enough on spatial intervention.
- The default map could already satisfy too much of the intended buildout.

**Why this misses the HW02 goal**

- If the player can succeed by waiting, then the project stops being a map-editing systems game.

**Upgrade plan**

- Keep the new first objective as a broken-route logistics repair objective.
- Keep the second objective as a resource refill objective that depends on the new network actually working.
- Keep the third objective as a fortify-and-stabilize objective so walls and spatial control matter.
- Extend the current broken-frontier scenario with more variants later, but keep the same rule: the player must reconnect a split logistics line rather than simply wait near the center.

### 5. AI Director & Policy Layer

**Current weakness**

- The earlier HW04 write-up treated the AI layer as a feature checklist item instead of a support layer for the core simulation.
- Group steering could distort local feasibility and produce misleading behavior.

**Why this misses the HW02 goal**

- The AI layer should explain and amplify system behavior, not replace the world simulation or invalidate it.

**Upgrade plan**

- Keep group policy as preference shaping, not direct control over local survival and delivery logic.
- Keep environment directives and policy outputs visible in the UI, but judge them by one standard: do they produce interpretable changes in traffic, roles, and pressure without breaking the base simulation?
- Continue using deterministic fallback as the reference baseline for regression, then compare live LLM behavior against that baseline.

### 6. Events / Weather / Ecology

**Current weakness**

- Weather and event pressure still act more like numeric turbulence than spatial gameplay in many runs.
- Predators, herbivores, traders, and saboteurs exist, but their relation to the player-built network is not always strong enough.

**Why this misses the HW02 goal**

- If events only move numbers, then they do not deepen the player's decisions about roads, storage, and defensive layout.

**Upgrade plan**

- Keep only the event and weather interactions that create readable map consequences.
- Keep localized weather hazard tiles so storms, rain, winter, and drought affect frontier routes and not just global multipliers.
- Keep spatial event payloads so bandit raids target concrete frontier zones, trade caravans reward reclaimed depot space, and animal migration pushes herbivores toward readable map destinations.
- Prefer threats that change route value, congestion, or safe build zones over purely scalar penalties.
- Tie walls and warehouse placement more directly to threat management so the player can see defensive geometry matter.

### 7. Interpretability

**Current weakness**

- The project has many debug panels, but not all of them help the player understand cause and effect.
- Some AI views report outputs that look authoritative even when the underlying behavior is weak.

**Why this misses the HW02 goal**

- The project promise is interpretable AI decisions. That means the debug layer has to help explain reroute and role changes, not just dump text.

**Upgrade plan**

- Keep telemetry that answers specific questions:
  - why did workers switch priorities?
  - why did routes change?
  - why did the objective stall?
- Keep the new explanation surfaces that answer those questions directly in moment-to-moment play:
  - HUD objective hints and frontier summaries
  - tile-level context in the Inspector
  - entity-level decision context in the Inspector and Entity Focus overlay
- Remove or deprioritize panels that do not help read those causal links.
- Treat the entity focus panel and AI trace as evidence tools, not decorative panels.

### 8. Performance / Deployment

**Current weakness**

- The build passes locally, but the previously published hosted URL was dead.
- The production bundle still emits a large chunk warning.

**Why this misses the HW02 goal**

- A system-heavy sandbox only matters if it can actually be run and verified by another person.

**Upgrade plan**

- Do not publish unverified hosted links in future reports.
- For this submission, treat the verified local build as the authoritative alpha artifact:

```bash
npm install
npm run dev -- --host 127.0.0.1 --port 4173
```

- Keep `npm test` and `npm run build` as mandatory validation gates for every milestone.
- If a new hosted deployment is added later, it must be checked on the same day it is reported.

## Release Notes Since HW03

### New

- Added a sparse alpha scenario that starts with one warehouse, two farms, one lumber site, minimal roads, and a partially damaged frontier.
- Replaced the old objective flow with a broken-frontier logistics repair objective, then stockpile recovery, then fortify-and-stabilize.
- Added an explicit west-route reconnection and east-depot reclaim requirement before the colony can scale up.
- Added spatial weather fronts and hazard tiles so route cost changes are localized and visible.
- Added spatial event payloads so bandit raids, trade caravans, and animal migration now target readable frontier zones.
- Clarified the in-game control contract so camera pan, build, and inspect no longer compete.
- Added interpretability summaries to the HUD, Inspector, and Entity Focus panels so players can see why a tile, route, or entity matters.
- Added global keyboard shortcuts for the core interaction loop: `1-6` build tools, `Esc` clear selection, `Space` pause/resume, and `Ctrl/Cmd+Z` undo/redo.
- Added snapshot camera persistence so saved views restore the same zoom and target instead of dropping the player into a different framing.

### Fixes

- Fixed the Active-state overlay so it no longer blocks pointer events during gameplay.
- Restored right-drag camera panning and moved inspect-only behavior to `Alt + Left Click`.
- Fixed reset/regenerate camera framing so the view returns to a reliable default instead of inheriting stale pan state.
- Fixed worker delivery precedence so carry can flow back to warehouses instead of stalling in a harvest loop.
- Sanitized fallback and runtime state targets so workers and traders are no longer pinned into sticky local action states.
- Prevented group policy from forcing non-hungry workers into survival states.
- Fixed event activation timing so spatial side effects apply as soon as an event becomes active.
- Fixed event and weather path pressure so frontier hazards affect pathfinding instead of only changing global numbers.
- Invalidated tile caches after scenario rewrites so warehouse and structure counts match the actual grid.

### Known Issues

- The current alpha is functional, but the scenario still needs stronger mid-game pressure after the first logistics buildout.
- The weather and event systems are now spatial, but they still need stronger mid-game escalation and more scenario variants beyond the current broken-frontier opener.
- `vite build` still reports a large chunk warning, although the build completes successfully.
- I did not include the previous hosted link because it was unreachable on March 10, 2026, and I could not verify a reproducible deployment target from this repository alone.

## Known Risks / Validation Gates

Every future revision of the alpha should pass the following checks before it is reported as playable:

- Input regression: after `Start Run`, the overlay must be hidden, the viewport center must hit the `canvas`, left click must select a tile or entity, and right drag must move the camera target.
- Economy regression: workers with carry must return to `deliver`, and policy/fallback steering must not override that local requirement.
- Gameplay regression: the default seed must not pass the main objective chain without meaningful map edits.
- Spatial pressure regression: weather fronts and active events must mark concrete frontier zones and change pathing or local behavior in those zones.
- AI regression: policy and state targets may bias coordination, but they may not override survival, delivery, or feasibility.
- Interpretability regression: the current objective hint, frontier summary, tile context, and entity decision context must agree with the world state they are describing.
- Build regression: `npm test` and `npm run build` must pass before submission.

## Stage Log

### Stage 1. Input / Game Loop / Interaction Polish

**Goal**

- Make the start-active-end flow, build/select/inspect loop, shortcuts, reset behavior, and snapshot recovery stable enough that a new player can operate the alpha without hidden rules.

**Problems found before work**

- The active overlay had already been fixed not to block hit testing, but Stage 1 still lacked a complete interaction pass.
- Session state was not modeled as a first-class saved runtime object, so phase-sensitive features were harder to reason about and validate.
- The game had no global keyboard shortcut layer for the main interaction loop.
- Reset, regenerate, and snapshot recovery did not fully stabilize camera framing, so the player could be left in an inherited pan state after a restart or load.
- UI messaging still relied too heavily on raw text, with weak severity cues for success versus errors.

**Iteration A fixes**

- Added a serializable `session` runtime state (`menu`, `active`, `end`) and normalized legacy snapshots during restore.
- Replaced the old local run-phase bookkeeping in `GameApp` with `state.session`, so pause, benchmark, render locks, and end-state checks now agree on the same source of truth.
- Added a global shortcut resolver for build tools and session controls:
  - `1-6` for tools
  - `Esc` to clear selection
  - `Space` to pause/resume while active
  - `Ctrl/Cmd+Z` and `Ctrl/Cmd+Shift+Z` / `Ctrl/Cmd+Y` for undo and redo
- Tightened selection behavior so tile selection clears stale entity focus and entity selection clears stale tile focus.

**Iteration B expansions**

- Added stronger UI feedback by surfacing action severity (`info`, `success`, `error`) directly in the HUD.
- Updated the overlay and build toolbar hints so the current control contract is visible in the interface instead of only in report text.
- Added snapshot camera persistence, so a saved view restores the same zoom and target when the snapshot is loaded.
- Added deterministic camera resets on regenerate/reset so the player re-enters a centered, readable framing instead of inheriting an arbitrary previous pan.

**Iteration C tuning and debugging**

- Fixed residual OrbitControls damping leakage during regenerate/reset/load by flushing stale pan inertia before finalizing the restored view.
- Re-tested pause/resume and undo/redo while the simulation was actively running, not only in unit tests.
- Cleaned the overlay copy so shortcut hints render as actual inline code instead of raw backticks.

**Tests added or changed**

- Added `test/shortcut-resolver.test.js` for digit tools, pause gating, selection clear, and undo/redo shortcuts.
- Added `test/hud-controller.test.js` so the HUD now verifies session phase text and action severity styling hooks.
- Extended `test/snapshot-service.test.js` to confirm session state and saved camera view metadata survive snapshot serialize/restore.
- Kept `test/game-state-overlay.test.js` as the overlay hit-testing regression for active play.

**Manual validation**

- March 10, 2026 local browser validation on `http://127.0.0.1:4173/` confirmed:
  - after `Start Run`, the overlay is hidden and the viewport center hits the `canvas`
  - right-drag changes the camera target
  - left-click selects/builds and `Alt + Left Click` inspects without building
  - `1-6`, `Esc`, `Space`, and undo/redo shortcuts work in active play
  - clicking a worker selects the entity and clears tile selection
  - saving a snapshot, resetting to menu, and loading the snapshot restores both the active phase and the saved camera framing
- Additional input smoke checks were run on three scenario families:
  - `temperate_plains`
  - `fortified_basin`
  - `archipelago_isles`

**Remaining risks**

- Small sub-tile camera drift can still appear immediately after a regenerate if the previous drag ended on the same frame; the reset is now stable enough for play, but this camera path could still be tightened further later.
- Stage 1 is stable, but it does not solve the deeper scenario-authoring and progression problems. Those remain Stage 2 and Stage 6 work.
