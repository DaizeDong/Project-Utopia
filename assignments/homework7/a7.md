## Engineering Retrospective

### Pillars (a1.md → ship)

The original a1.md committed to three CG pillars: **NPC AI**, **Boids**, and
**A\* Pathfinding**. The HW7 ship treats Boids as a sub-system of pathfinding
(separation/alignment/cohesion are dampened to 0.35× when an entity is on an
A\* path, so convoys flow rather than scatter), so the talk and this report
cover two pillars.

**Pillar 1 — NPC AI.** I shipped a flat **PriorityFSM** dispatcher in 125 LOC
(`src/simulation/npc/PriorityFSM.js`) that hosts both the Worker FSM (12 named
states, 38 priority-ordered transitions) and the Visitor FSM (9 states). The
spec called for Behavior Trees, but during v0.10.0 I rewrote the worker AI
twice — first to a utility scorer with hysteresis (v0.9.0), then to the flat
priority FSM that actually shipped (v0.10.0). FSM was the right shape because
the question we ask each tick is "what events can preempt the current state?"
not "what action under conditions A/B/C?" — the same intent expressed more
cleanly. Combat preempt fires at `priority: 1`, naturally beating
`DELIVERING`'s `priority: 5` without a special-case `isSurvivalCritical`
bypass. The LLM (`gpt-5.4-nano` via direct OpenAI) emits *intents* into a
queue; the FSM picks up the work; a deterministic rule-based fallback
director emits the same plan-shape so the game runs without network. Net
delta from the rewrite: **−2530 LOC across 35 files** (plan target was
−1300; over-delivered 2.0×). The Inspector panel surfaces FSM state, intent,
hunger, and target — every agent is fully observable.

**Pillar 2 — Pathfinding & Navigation.** Grid is 96 × 72 = 6912 tiles, 14
tile types. A\* is 8-neighbor cost-weighted; Boids is layered on top with the
**separation dampening** trick described above. The interesting problem is
**bridges across water**: when should the colony build one? I shipped a
**dual-search A\*** that runs the search twice (`allowBridge: false` and
`true`), scores both with `length × TRAFFIC_AMORTIZATION` (50 round trips),
and only commits to building the bridge when the amortized win clears the
build cost. A 3-tile bridge that saves 20 detour-tiles wins; a 1-tile bridge
that saves 2 detour-tiles loses, and workers walk around. The
BridgeProposer scans 8-tile shoreline pairs; planning composes with a
fog-aware build gate so IDLE workers wander toward the fog edge to scout
when build candidates are hidden. Combined with the dampened-separation
convoy behaviour, the visible result is workers flowing in single file along
roads, predators scattering organically, and bridges appearing at the
shorelines that actually deserve them.

**Cool result.** The two pillars are not separate features — the same loop
crosses them every couple of seconds: LLM (Pillar 1) requests a quarry →
BuildProposer (Pillar 2 fog gate) latches `scoutNeeded` → IDLE workers bias
toward fog edge (Pillar 1+2) → stone revealed → BUILDER FSM seeks site (Pillar
1) → A\* path with Boids on top (Pillar 2) → predator appears → COMBAT_PREEMPT
fires at `priority: 1` (Pillar 1) → resume path on resolve (Pillar 2).

**Engineering scale at ship (HEAD `9c7ed5a`):** 13 review/plan/implement/
validate rounds (R0 → R13), ~140 commits across the loop, **2060 tests pass /
0 fail / 4 skip**, **+395 tests added** since the R0 baseline of 1665, and
**7 consecutive rounds with 0 test failures** (R7 → R13). Long-horizon
benchmark (seed=42, plains, 90 days): DevIndex went from the HW6 baseline of
**37.77 → R8 peak of 73.18 (+93.7%)** with full 90-day colony survival; R12
day-30 confirmed parity at 72.61.

### Technical Post-Mortem

The full Post-Mortem is at
[`assignments/homework7/Post-Mortem.md`](./Post-Mortem.md). The summary:

- **Spec → ship pivot — BT to FSM.** a1.md committed to Behavior Trees with a
  shared blackboard. Built it, then realised BT is the wrong shape for our
  preempt-heavy problem (a HARVESTING worker who sees a saboteur 2 tiles
  away needs to switch *now*, not wait for the harvest tree to short-circuit
  every leaf). Flat priority FSM expresses preempt as a single `priority: 1`
  transition. The intent — autonomous, interpretable agents — held; the
  expression changed.
- **Two D-grade systems still in the codebase.** `ColonyPlanner` (1884 LOC)
  and `ColonyPerceiver` (1970 LOC) are functioning code but architecturally
  patch-piles. The migration to a `BuildProposer` interface (R5 + R6)
  extracted the safety-net logic, but the planners themselves were not
  decomposed before the freeze landed. Both are documented as HW8 candidates
  in Post-Mortem §6.
- **Hard freeze in the polish phase.** From R5 onward the loop forbade new
  tiles, roles, buildings, mood mechanics, audio assets, and UI panels.
  This is why the ship has no audio (substituted with toast notifications
  for raid / death / milestone events) and no walk-cycle animation
  (minimalist sphere-on-grid was an a1.md commitment anyway, and it scales
  to 200+ entities at 60 fps because rendering is one InstancedMesh per
  role). Audio + walk-cycle are documented deferrals.
- **Real Chrome FPS measurement deferred.** Headless Playwright's RAF cap
  (~30 fps) masked the actual 60-fps render budget for two rounds (R2 + R3)
  before I figured out that I was measuring my own measurement instrument.
  The fix was to gate the perf-bench on `process.env.CI !== 'true'` and run
  the headed-browser smoke separately. The lesson: telemetry first,
  optimisation second — measure your measurement before measuring your
  subject.

### AI Tool Evaluation

> Following the TA's anti-LLM-polish guidance (Post-Mortem §1.5: "this section
> must be in your own voice"), the evaluation below is first-person prose I
> wrote myself, not regenerated from a prompt.

I ran HW7 as a 13-round orchestrated review-plan-implement-validate loop
backed by Claude Agent SDK + Playwright headless browser. Three things
changed about how I worked:

**The wins.**

- **Speed of iteration.** 13 rounds, ~140 commits, 2060 passing tests, in
  roughly two weeks of wall-clock. That's 10×–20× of what I could do
  manually. The leverage is real, and the leverage is in *iteration* (cycle
  time of "I have an idea → it's tested → I've decided whether to keep it"),
  not in raw lines of code per minute.
- **Quality of self-criticism.** Letting one subagent review code another
  wrote caught issues I would have rationalised away. The single most
  valuable example: in R7 a perf reviewer agent flagged a **latent
  ReferenceError** in `WorkerAISystem.js` — a constant called
  `WORKER_EMERGENCY_RATION_HUNGER_THRESHOLD` was referenced but never
  declared. The path was unreachable in the current state, but if I'd ever
  flipped a feature flag, instant crash. The agent caught it because it
  noticed the symbol wasn't in the file's exports while doing an unrelated
  perf trace. I would not have noticed.
- **Cross-round bisection.** In R11 an audit agent **bisected a 4-commit
  regression** I didn't know existed — ran the long-horizon benchmark at
  every commit between R8 (peak DevIndex 73) and R10 (DevIndex 67), found
  the −60% drop, and traced it to a single line of phase-offset math in the
  cascade-mitigation fix. One-line revert recovered 91% of the bench.

**The costs.**

- **Self-inflicted regression caught 3 rounds late.** In R5 I added a
  4-second `roleChangeCooldownSec` to prevent worker role-thrashing.
  Three rounds later, a playtester reviewer reported "BUILDER never claims
  construction sites; blueprints sit unbuilt forever." The cooldown was
  blocking FARM→BUILDER promotion. The fix was a one-line predicate:
  bypass the cooldown when sites are unclaimed. The lesson is uncomfortable:
  **self-imposed safeguards can become the bug, and agents trust your
  prior commits as ground truth** — they don't auto-question your past
  fixes. I now anchor each rollback target with a named SHA so cross-round
  bisection has somewhere to land.
- **Headless RAF chasing.** Two rounds (R2 + R3) lost to chasing FPS bugs
  that turned out to be Playwright's headless RAF cap, not a real
  rendering regression. The cost was real wall-clock; the fix was
  embarrassingly small. Telemetry trust matters.
- **Hard guardrails matter more than clever prompts.** The frozen feature
  scope, the named rollback anchors (`v0.10.1-h`, `v0.10.1-i` etc.), the
  file-ownership tracks (one subagent owns ≤ 3 files per round) — those
  are what turned the agents from creative-but-flaky into trustworthy.
  Without those constraints the agents drifted; with them they shipped.

**Honest framing.** The LLM did not write the architecture (PriorityFSM
design, dual-search bridge interleave, separation-dampening-on-path were all
hand-designed). The LLM did write large amounts of glue code, contract
tests, and refactor passes — and it caught two latent bugs and one
cross-round regression I would have missed. The ratio of "agents wrote it" to
"I designed it" is roughly 70 / 30 by line count and roughly 30 / 70 by
load-bearing decision. That ratio is the part I'd most want a future
employer to understand.

## Final-Polish-Loop Process (companion to peer playtest)

HW6's a6.md substituted 10 LLM agents for 5 human classmates and ran a
9-round Reviewer / Enhancer / Implementer / Validator pipeline. HW7 inherited
the same shape and added **specialised reviewer roles** (A1 stability hunter,
A2 perf auditor, A3 first impression, A4 polish/aesthetic, A5 balance critic,
A6 UI design, A7 rationality audit, B1 action-items auditor, B2 submission
deliverables, C1 code architect) plus **playability reviewers** (PA through
PII, one per round R5–R11) that scored a 1–10 holistic playability rank.

The full audit trail is checked in at
[`assignments/homework7/Final-Polish-Loop/`](./Final-Polish-Loop/) — every
round has `Feedbacks/`, `Plans/`, `Implementations/*.commit.md`, and
`Validation/` directories, and `PROCESS-LOG.md` carries the round-by-round
closeouts. Holistic playability rank trajectory: **R5 5.5 → R8 4.0 (low,
during structural rebuild) → R11 5.5 (back to baseline)**.
