What it is.

Premise

  • Player builds infrastructure and sets priorities
  • Workers, visitors, predators and herds execute autonomously on a 96 by 72 tile grid (about 6900 tiles)
  • Every living entity is a simple sphere; the world is tile-based; the aesthetic is intentionally minimalist
  • The system reacts; you steer

Why simple spheres?

The goal is to make autonomous behavior visible and interpretable. Reducing visual complexity lets you watch flowing convoys, scattering herds, and traffic patterns reshaping as you build.

Pillar 1: how an agent decides.

The challenge

Each agent in the world (workers, visitors, animals) must:

  • Continuously evaluate its own state (hunger, carry, role, surroundings)
  • Choose among many behaviors (harvest, deliver, build, eat, fight, rest)
  • React to global colony state (food shortage, raid, blueprint placed)
  • Be interpretable: click any agent and see why it is doing what it is doing
Original proposal: a Behavior Tree, a tree of conditional checks with a shared blackboard.
What shipped: a flat Priority State Machine, a small set of named states with priority-ordered transitions.
Same intent, better fit. Reasoning on the next slide.

Priority State Machine, plus a Large Language Model on top.

Why a state machine, not a tree

  • A Behavior Tree asks: "what action under these conditions?"
  • A state machine asks: "what transitions can fire from this state?"
  • Our problem is transition-driven. A worker harvesting wood must preempt instantly when a saboteur appears two tiles away.
  • The state machine expresses preemption as one priority-1 transition: a single line. A tree would need an interrupt check at every node.

Twelve named worker states

IDLE → SEEKING_HARVEST | SEEKING_BUILD | FIGHTING | RESTING
SEEKING_HARVEST → HARVESTING | FIGHTING (preempt)
HARVESTING → DELIVERING | FIGHTING (preempt)
DELIVERING → DEPOSITING | IDLE
SEEKING_BUILD → BUILDING | IDLE
BUILDING → IDLE | FIGHTING (preempt)
FIGHTING → IDLE  // always settles back

Two-layer architecture

┌──────────────────────────────────────┐ │ strategic director │ │ · uses a Large Language Model │ │ · reads world state │ │ · emits a high-level plan │ └────────────────┬─────────────────────┘ │ intents ┌────────────────▼─────────────────────┐ │ priority state machine │ │ · per-worker execution │ │ · picks nearest available task │ └──────────────────────────────────────┘
  • The language model never moves a worker directly. It only changes priorities.
  • When the language model is offline, a deterministic fallback director emits the same plan shape from rule heuristics. The game runs cleanly without network.
  • Over 2000 automated tests cover the AI without any language model calls.

Pillar 2: how groups move together.

The challenge

Living things move in groups, all on the same map at the same time:

  • Dozens of workers walk the roads carrying resources
  • Herds of deer roam the fields
  • Packs of wolves stalk the herds
  • Visitors and traders enter and leave the colony daily

The motion must be

  • Locally avoidant: agents do not bump into each other
  • Globally coherent: a herd stays a herd
  • Reactive: predators chase, prey flees, workers form convoys

Boids: three simple rules, one weight per context.

The three Boids rules

Boids is a classic flocking simulation introduced by Craig Reynolds in 1986. Each agent follows three local rules:

  • Separation: steer away from the closest neighbors so agents do not collide
  • Alignment: match heading with neighbors so a flock points the same way
  • Cohesion: drift toward the average position so a flock stays together
  • Plus a fourth pull: path-following toward the next tile of a planned route
The wrinkle: pure Boids on a worker convoy fights the planned route. Separation pushes workers apart; path-following snaps them back. The visible result is jitter.

Fix: context-aware separation weight

const sepWeight =
  (entity.path && entity.pathIndex < entity.path.length)
    ? 0.35  // worker on a path: dampen separation
    : 1.0;  // animal or no path: full Boids
  • Workers in transit form flowing convoys, visibly single-file along roads
  • Animals such as deer and wolves keep full Boids and scatter organically
  • Same Boids implementation hosts both behaviors. One weight switches between them.

Pillar 3: getting from A to B.

The challenge

Workers walk on a 96 by 72 grid (about 6900 tiles). The path must:

  • Avoid impassable terrain (water without a bridge, walls)
  • Honor realistic costs (mountains slow, roads fast, bridges expensive but valuable)
  • Avoid collisions with workers and animals without breaking the path
  • Adapt to fog of war: the colony cannot build on tiles it has not yet revealed
Two layers, classical: A-star (a global pathfinding algorithm) for the route, plus Boids for local steering. The interesting work is where the two layers meet.

Dual-search A-star with amortized bridge cost.

The bridge problem

  • If water is impassable, the planner refuses to cross rivers, even when a 3-tile bridge would save 20 tiles of detour
  • If water is just a low cost, the planner routes through water everywhere, oblivious that bridges are not built yet

Solution: run A-star twice

Once disallowing bridges, once allowing them. Score by cost amortized over expected traffic. Each route is assumed to see roughly 50 round trips before becoming obsolete.

const noBridge   = aStar({ allowBridge: false });
const withBridge = aStar({ allowBridge: true  });

const TRIPS = 50;  // expected round trips
const sNo   = noBridge.length;
const sWith = withBridge.length +
              (withBridge.bridgeCount × BRIDGE_COST) / TRIPS;

return sWith < sNo ? withBridge : noBridge;

Result

  • A 3-tile bridge that saves 20 detour tiles times 50 trips: clear win, build it
  • A 1-tile bridge that saves 2 detour tiles: not worth it, walk around
  • The bridge proposer scans 8-tile shoreline pairs and can propose multi-tile bridge sequences, not just 1-tile pinch points

Plus: fog-aware planning

  • The colony director rejects fogged tiles for new blueprints
  • Idle workers bias their wander toward fog edges to scout
  • "Build a quarry near stone" cannot succeed until workers reveal where stone is

What I learned.

Three lessons

  • Spec fidelity is not the goal; design fidelity is. The original proposal said Behavior Tree, what shipped is a Priority State Machine. Same intent, better fit. The intent of "autonomous, interpretable agents" is what mattered; the data structure followed.
  • Telemetry first, optimization second. I lost two rounds chasing performance bugs that turned out to be a measurement quirk in the headless test browser. Measure your measurement before measuring your subject.
  • AI agents need hard guardrails, not clever prompts. Across thirteen rounds and 140 commits, what made agents trustworthy was constraints (frozen scope, rollback points, per-track file ownership), not prompt cleverness.

Insights from AI-assisted development

  • The Large Language Model was best as a reviewer, not a writer. Letting one agent design, another implement, and a third audit caught issues a single agent would have rationalized away.
  • Most surprising: an agent bisected a four-commit regression. It ran benchmarks at every commit between rounds eight and ten, found a 60% drop in our composite metric, and traced it to a single line of phase-offset math.
