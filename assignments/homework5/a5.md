# Project Utopia HW05 (Beta Build)

Daize Dong, NetID dd1376

## Quick Review: First Pillar and Current Achievement

The first technical pillar of Project Utopia is **NPC AI / Behavior Trees with runtime LLM assistance**. By HW03, the project already proved that an LLM-backed environment director and policy system could run in real time with schema validation and deterministic fallback. By HW04, that AI layer had been placed inside a complete alpha loop with menu, active simulation, and end-state flow. In this HW05 Beta build, I kept that AI layer intact and focused on making the rest of the project feel like one coherent product instead of a collection of independent systems.

## Assignment 1-4 Continuity

- **HW01:** I selected three pillars: `NPC AI / Behavior Trees`, `Swarm & Boids Modeling`, and `Pathfinding & Navigation`.
- **HW02:** I defined the technical goal as a browser-based Three.js simulation that should stay near `60 FPS` at roughly `200 total entities`, while integrating runtime AI plus deterministic fallback.
- **HW03:** I proved the main AI chain worked and could fall back safely.
- **HW04:** I built the alpha wrapper: full run loop, objective chain, HUD, panels, scenario pressure, and stability tooling.
- **HW05:** The Beta goal is not to throw away that structure, but to deepen the interaction between the secondary pillars and improve visual cohesion.

## Secondary Pillar Integration

The second-stage pillars are now more deeply connected to the visible gameplay loop instead of living mostly in debug text:

1. **Pathfinding & Navigation**
   - Broken frontier routes, depot targets, traffic hotspots, and weather fronts now show up directly on the map through a pulsing pressure-lens overlay.
   - This makes route repair and logistics decisions readable in the world itself rather than only through the inspector or telemetry.

2. **Swarm / Boids / Ecology Pressure**
   - Wildlife pressure hotspots are now visualized in the same map-space language as logistics and weather pressure.
   - Herd motion, route congestion, and ecological pressure now feel like interacting systems competing for the same frontier space.

3. **Visual Cohesion**
   - The renderer now derives atmosphere from `scenario family + weather + spatial pressure + session outcome`.
   - Instead of a mostly static bright prototype look, the scene now shifts mood between frontier, chokepoint, and island scenarios, while storms, droughts, and end states visibly change the world tone.

## Beta Release Notes (Since HW04)

### New Features

1. Added a **scenario-aware atmosphere system** that changes sky/fog/light response based on scenario family, weather, pressure, and win/loss state.
2. Added a **pressure-lens map overlay** that highlights unresolved route gaps, depot goals, active event zones, weather fronts, traffic hotspots, and ecology hotspots.
3. Upgraded the **start briefing** into a Beta-style mission panel that summarizes the scenario, current objective chain, AI/pressure state, and the original project pillars.
4. Upgraded the **end-state overlay** into a clearer Beta run summary with scenario/objective framing instead of a minimal alpha popup.
5. Added renderer-side helper modules and tests so the new visual systems are isolated, deterministic, and less likely to break gameplay logic.

### Secondary Pillar Integration Summary

The core AI pillar still decides high-level intent, but now the consequences of those decisions are easier to see spatially. A* routing pressure, Boids congestion, wildlife grazing pressure, and scenario weather fronts all feed the same visible map language. This means the user can interpret why the colony is struggling or succeeding without needing to read only the debug dock.

### Performance Check

My HW02 technical target was to stay around `60 FPS` in the browser with roughly `200 total entities`. For this Beta iteration, I intentionally kept the new work mostly inside the renderer and UI layer:

- Atmosphere changes are derived from existing state and only lerp a small number of light/fog values.
- Pressure-lens recomputation is signature-gated so the expensive marker derivation only reruns when relevant world state changes.
- The simulation logic itself was not expanded with a new heavy system in this pass.

Validation run on the latest `main` worktree:

- `npm test` -> pass
- `npm run build` -> pass

## Known Issues

1. The production bundle still contains a large `vendor-three` chunk. This does not block the build, but it is still an optimization target before the final submission.
2. Live-LLM validation still depends on a valid local API key and reachable proxy. The project remains safe in deterministic fallback mode when that is unavailable.
3. The new pressure-lens markers are intentionally descriptive rather than interactive; they improve readability, but they do not yet support click-through inspection by themselves.
