# 02c-speedrunner

评测日期：2026-04-23  
评测入口：`http://127.0.0.1:4173/`  
评测身份：硬核 min-max / speedrunner / 找 cheese 的玩家

以下结论**只来自本轮浏览器构建体验**。我没有参考源码、文档、历史轮次、其他反馈，也不会用那些信息修正这里的判断。

## 身份自述

我这类玩家进入 Project Utopia，不是为了“随便玩一局看看气氛”，而是会立刻问三个问题：

1. 开局最短路径是什么。
2. 有没有一条明显高于其他路线的 dominant strategy。
3. 游戏有没有能力承受高频 restart、极端暂停操作、快速试错和数值压榨。

因此我对首屏、信息架构、开局读秒感、重开效率、以及“点下开始以后能不能稳定进入可操作局面”非常敏感。对一个打策略深度和长期存活的作品来说，真正的第一道门槛不是后期平衡，而是：**我能不能在 10 秒内开始第一轮实验，并在 30 秒内判断这局值不值得继续。**

这次浏览器体验里，我能明确观察到的是：标题区包装、模板入口、地图尺寸、快捷键提示和“Start Colony / How to Play / New Map”三按钮结构已经具备“准备开跑”的味道；但最关键的那一步，也就是从准备界面切到实际 run，稳定性没有给出足够的正反馈。对 speedrunner 视角来说，这不是小瑕疵，而是会直接压垮整个评测的基础问题。

## Run 1 日志

### 进入前可见信息

初始界面给出的信息包括：

- 标题：`Project Utopia`
- 场景描述：`Reconnect the west lumber line, reclaim the east depot, then scale the colony.`
- 当前模板默认值：`Temperate Plains`
- 地图尺寸默认值：`96 × 72`
- 快捷提示：RMB 拖拽预览、滚轮缩放、`1-12` 工具、`Space` 暂停、`L` heat lens、`F1` 帮助
- 目标/状态文案可见：`Survive as long as you can`、计时 `00:00:00`、分数 `0 pts`

从 speedrunner 角度，这个首屏有两点是对的：

- 它没有把我扔进冗长剧情或大段说明，理论上可以直接开跑。
- 它把高频键位提前露出来，说明设计者知道玩家会追求操作密度和信息压缩。

但也有明显问题：

- 现在能看到的目标更像一句氛围化口号，而不是一条可优化的开局任务链。
- `Dev Index`、威胁值、工人死亡、资源曲线这些 speedrun 关键指标在首屏并未形成很强的“可测量承诺”。
- 模板有 6 个，但没有任何预估标签，例如资源稀缺度、防守压力、节奏倾向、推荐打法。对反复开图的人来说，这会增加试错成本。

### 开始操作

我点击了 `Start Colony`。

### 结果

这一步之后，评测会话没有继续给出可稳定读取的游戏内操作状态。换句话说，本轮浏览器体验里，**从准备界面进入正式 run 的过程没有形成一个可持续评测的可操作回合**。对我而言，这等同于：

- 无法确认实际地图是否成功载入；
- 无法记录 run 内资源曲线；
- 无法验证暂停、多建造、快节奏重开等 speedrunner 核心操作；
- 无法检查数值 cheese、上限策略、或后续崩盘模式。

对“想跑分的人”来说，开局切场失败或不稳定，本身就是致命结论：它阻断了实验循环。

## Run 2 日志

本轮没有完成第二次有效 run。原因不是我主动停止实验，而是当前可观察构建在第一次进入 run 后，已经没有继续给出稳定、可复用的交互回路。

从评审标准看，这意味着两件事：

1. **重开效率无法验证。**  
   speedrunner 非常依赖“开图 - 判断 - 放弃 - 再开图”的短循环。如果连第一轮进入都不稳定，那么“多次重开比较种子/模板”的玩法价值就没有落地。

2. **模板差异无法验证。**  
   尽管菜单里能看到 `Temperate Plains / Rugged Highlands / Archipelago Isles / Coastal Ocean / Fertile Riverlands / Fortified Basin` 六种模板，但当前浏览器体验没有让我实际跑起来比较这些模板是否存在明显 meta 分化。

如果一个游戏把策略深度主要押在模板差异与中后期运营上，那么进入 run 的稳定性就是整个设计被看见的唯一入口。这个入口当前没撑住。

## Run 3 日志

同样没有形成第三次有效 run。

需要说明的是，这不是“我时间不够，所以没跑够三次”；而是我在本轮可见构建里，**没有拿到可以持续推进第二、第三轮实验的稳定局内状态**。因此我不能诚实地编造分数、存活时间、死亡人数、Dev Index 或任何数值表。

从盲审角度，宁可明确写“跑不起来/无法完成有效回合”，也不能把未观察到的系统说成已经验证过。  
所以这里的结论非常直接：

- 当前浏览器版本，至少在我这次评测中，**首个 run 入口已经足以阻断深度评估**。
- 对 speedrunner 玩家而言，这比“后期平衡偏斜”更严重，因为它让后面的所有优劣都失去被检验的机会。

## 找到的策略与 cheese

严格说，这次我**没有机会验证真正的 cheese**，因为 run 没有稳定展开。但从 speedrunner 视角，已经可以指出几个很关键的风险点：

### 1. 最大的“反策略点”是入口稳定性

如果“开始殖民地”这一步本身不能稳定进入回合，那么最优策略不是游戏内策略，而是“继续尝试把局打开”。  
这会把本该发生在资源、建筑、道路、防御、工人分配上的优化竞争，退化成对程序入口稳定性的碰运气。

### 2. 首屏提供了很多快捷键，但没有证明这些键位真的构成高水平玩法空间

可见键位包括 `1-12`、`Space`、`L`、`F1`、滚轮、RMB 等。  
这说明设计方向是想做成一个支持高操作频率的管理/建造游戏，这是对的。问题在于：

- 玩家还没来得及把这些操作转化为收益；
- 就目前可见体验，系统先展示了“你可以很快”，但没有兑现“你快起来以后游戏能接住你”。

### 3. 模板选择有潜在 meta，但当前版本没有把它兑现为可测对象

六个模板理论上已经足够做出明显的开局倾向差异，例如：

- 木材优先图
- 食物压力图
- 岛屿物流图
- 高防守压力图
- 高成长图

但如果缺少稳定的进入与重开回路，玩家就无法把这些模板差异转化为可比较的胜率、节奏或分数策略。  
对 min-max 玩家来说，这意味着“潜在深度存在，但被交互稳定性锁死了”。

## 游戏有没有真正的玩法上限

就这次浏览器构建给我的结论：**我暂时看不到已经被兑现出来的玩法上限，只能看到它想做上限。**

为什么这么说：

1. 首屏信号说明作者是有上限设计意识的。  
   有模板、有尺寸、有工具热键、有暂停、有热力镜头、有“尽量活久”的跑分目标，这些都不是纯 casual 布置。

2. 但我没有在运行层面看到这些系统被串成“可反复练、可比较、可压榨”的 loop。  
   真正的玩法上限，不是功能列表，而是玩家能否稳定地进入同一个规则容器里反复优化。

3. 当前最突出的不是“哪条路线最强”，而是“能否正常开始并持续一局”。  
   只要这一点没稳住，讨论 wood-first、food-first、road-first、wall-spam、warehouse-spam 哪条更强，都还太早。

所以我的判断是：

- **设计意图上，游戏是朝着有玩法上限去的。**
- **产品完成度上，这个上限目前还没有在浏览器体验里被可靠地交到玩家手里。**

## 具体问题总结

### A. 入口稳定性不足，直接毁掉 speedrunner 体验

这是本次最严重问题。  
策略游戏允许平衡还粗糙，但不允许“开始实验”这一步不稳定。因为硬核玩家的整个乐趣，来自几十次、几百次短局实验叠加出来的理解。

### B. 首屏信息偏“会玩的人看得懂”，但离可量化还差一截

现在的提示更偏操作说明，而不是实验说明。  
如果目标用户包含 speedrunner 或 min-max 玩家，建议在开始前就告诉我：

- 模板的核心压力是什么；
- 是否有 seed；
- 分数由什么构成；
- Dev Index 与存活/损失/威胁的关系是什么。

这些都不用写成长教程，只要信息可扫描、可比较即可。

### C. “可以快玩”与“值得快玩”之间还有断层

你已经给了我很多热键和高频操作入口，这是一种正确姿态。  
但 speedrunner 真正在意的是：我多快地做这些动作，会换来多大的回报差异。  
这部分当前在浏览器体验中没有被证明出来。

## 评分与一句话总结

**评分：2/10**

一句话总结：**首屏有明显的硬核经营/跑分设计企图，但这次浏览器构建在最关键的开局进入阶段就没能提供稳定可持续的 run，因此 speedrunner 无法真正开始做策略优化。**
