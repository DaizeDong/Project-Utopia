# Round 8 盲测反馈：02c-speedrunner

## 总评分（10 分制）

**6.0 / 10**

从速通 / min-max / 优化玩家视角看，这个 build 已经有明确的资源链、路网、工人状态、Dev Index 与快进控制，表面上可以做“开局路线优化”。但目前最强策略不是精细经营，而是尽快利用免费道路、暂停建造与自动 AI 的高容错，把物流距离压低后等系统自己跑。食物压力很快变成唯一硬门槛，其他资源与建筑路线的策略差异被饥饿死亡、寻食优先级、8x 下的 AI 抖动吞掉。它有可优化空间，但还没有形成稳定、可复盘、可挑战的玩法上限。

## 身份自述

我是 `02c-speedrunner`，这次只按盲测 reviewer 的身份玩当前 build，不看源码、不看 README、不看历史反馈、不看 git log/diff，也不读取旧分数或作者意图。我把自己当成一个想“最快打穿目标、找 dominant strategy、找 cheese、找数值漏洞”的玩家，而不是普通体验玩家。

我的评测原则是：如果一个策略可以无脑复制，并且比其他策略更稳、更快、更少操作，那它就是 dominant；如果暂停、快进、免费建筑、AI fallback 或 UI 逻辑能让风险消失，那就是 cheese；如果同一开局重复几次结果差异很大，但玩家不能解释原因，那就是可控性不足。

## Run 日志

### Run 0：接管时的手动道路局

接管页面时，殖民地已经在一个相对发育的状态：屏幕显示 `Survived 00:03:22` 到 `00:03:23`，资源大约为食物 216/217、木材 7/11、石头 0、草药 5、人口 32，底部显示 `Dev 60 · thriving`。Autopilot 是 off，手动控制开启。

地图中心已经有大量道路，西侧林区和东侧 ruined depot 都被道路连上，物流基本打通。工人面板里大部分人在 Deliver / Seek Task / Seek Food 之间切换，角色覆盖 FARM、WOOD、STONE、SMITH、HERBS、COOK、HAUL 等。这个状态给我的第一印象是：只要把道路铺出去，AI 会自己接管很多细节，玩家并不需要逐个调度。

这一局最像“速通正确答案”的雏形：先修路，压缩路径，再让自动职业系统吞掉采集与交付。Dev 到 60 后提示 meals flowing、建议 Smithy，说明经济已经进入可持续阶段。但问题是，玩家很难看出这个成功到底来自道路、农田、厨房、仓库，还是 AI 自动分配本身。作为优化玩家，我希望能知道每个建筑动作带来多少节省或增益，但当前反馈更偏叙事，不够数值化。

### Run A：Autopilot + 8x，无手动优化

我重开后尝试打开 Autopilot，直接切到 Ultra speed 8x，让系统自己跑，尽量不做手动建造。约 30 秒真实时间后截图 `runA_autopilot_8x.png` 显示：屏幕顶部 `Survived 00:01:16`，`Score 106`，`Dev 51/100`，资源约食物 25、木材 0、石头 0、草药 1、人口 25。顶部死亡日志显示有工人 starvation，说明已经从初始人口明显掉人。

这一局的关键观察是：Autopilot 很快能铺出西侧 lumber route，并开始高级生产，但它不能稳定避免食物崩盘。死亡发生得很早，且死因是饥饿，不是外部威胁。换句话说，自动 AI 的 early-game build order 对速通玩家来说不够可信：它会推进路线与生产，但没有把“最低食物安全线”当成硬约束。

不过，Autopilot + 8x 仍然很强，因为它的操作成本极低。即使死了几个人，Dev 仍能到 51，Score 106，且系统持续运转。对于 min-max 玩家来说，这说明死亡惩罚可能不够尖锐：如果死 7 人左右仍能拿到不错的 Dev/Score，那“牺牲人口换发展速度”可能成为隐性策略。

### Run B：暂停规划 food-first + 8x

第二个主动策略是暂停后手动规划 food-first：先点 Farm，在中心和道路附近连续尝试铺多块农田；再点 Kitchen 放在中心附近；然后用 Road 把中心、东侧 depot、西侧资源点串起来。之后恢复并切 8x。暂停阶段截图为 `runB_paused_food_first_plan.png`，运行后截图为 `runB_food_first_8x.png`。

结果并不理想：截图显示 `Survived 00:01:16`、`Score 106`、`Dev 51/100`，资源约食物 19、木材 8、石头 0、草药 1、人口 8。屏幕中央出现 `Tova Tull starved`，顶部日志也显示 woodcutting specialist died of starvation。更糟的是，底部实际速度只有 `actual x4.0` 左右，而不是稳定 x8。

这个结果暴露两个问题。第一，food-first 的手动规划没有明显优于自动开局，甚至人口掉得更惨。可能原因是建筑落点无效、资源不足、工人建造/收获优先级不对，或者暂停规划后恢复时 AI 仍然无法优先完成食物闭环。第二，玩家在暂停状态下可以大量下达建造意图，但系统对“哪些能建、哪些缺资源、哪些会堵物流、哪些只是排队空转”的即时反馈不够强。

从优化角度看，这会让玩家很难形成稳定路线。一个优秀速通开局应该能被描述成“第 0 秒暂停，放 A/B/C，开 8x，1:00 达到 X”。但这里我的 food-first route 和 autopilot route 最终显示了几乎相同的 Survived/Score/Dev，却有完全不同的人口残局，这说明可解释性和可重复性还不足。

### Run C：道路/仓库/墙体 spam 与 8x 压测

我尝试做第三类边界：暂停状态下大量 Road、Warehouse、Wall spam，再 8x 快进，看系统是否允许免费路网、低成本屏障、仓库缓冲形成 cheese。这个过程中浏览器/CDP 响应明显变慢，第三组没有稳定产出最终截图；但从前两局和接管局已经能看出道路是最接近 dominant 的建筑：Road 免费、连接规则宽、物流收益直接，且没有明显维护成本或拥堵惩罚。

墙体和仓库的压测没有完整完成，所以我不把它作为定量结论。但暂停批量下单本身是一个需要限制的点：如果玩家能在时间冻结时随意规划长距离物流和防线，再一键 8x 执行，真正的压力会被“暂停规划”削弱。除非游戏本来就是规划解谜，否则建议让暂停建造至少有更强的预览、资源锁定、队列上限或执行风险。

## 找到的策略与 cheese

第一，**免费 Road 是当前最强候选 dominant strategy**。道路不消耗资源，却直接缩短交付、采集、维修和探索路径。只要连接规则允许，玩家自然会倾向于尽可能早、尽可能多地铺路。道路的机会成本主要是工人施工时间，但在暂停规划和 8x 下，这个成本的体感很低。

第二，**暂停规划 + 快进执行有 cheese 倾向**。暂停时可以慢慢找格子、点大量建筑、规划路线，然后 8x 让系统跑。若没有队列上限、资源预占、错误惩罚或事件压力，熟练玩家会把实时压力完全转化为离线规划，生存游戏的紧张感会下降。

第三，**死亡可能被当作可接受成本**。Autopilot 局已经死了人，但 Dev/Score 仍然可观。如果最终评分更看重 Dev 而不是人口质量，那么“让边缘工人饿死，核心生产继续推进”可能成为邪道。

第四，**手动 food-first 目前不明显优于 Autopilot**。这不是说 food 没用，而是玩家投入的操作没有得到清晰收益。一个 min-max 玩家发现手动规划更麻烦、结果还不稳定，就会回到 Autopilot + Road 的低成本路线。

第五，**东侧 ruined depot 的路线目标很诱导**。接管局能看到道路打到 east ruined depot 后，经济明显更稳；Run A/B 也都出现 east depot 标记。速通路线大概率会围绕“先连 depot，再补食物/厨房”展开，而不是多种路线并存。

## 数值 / 快进 / AI 问题

食物数值压力过尖。Run A 和 Run B 都很快出现 starvation，且发生时玩家很难判断是农田太少、厨房太少、工人路程太长、AI 没优先吃饭，还是食物生产转换太慢。食物是硬门槛没问题，但现在它像黑箱惩罚。

8x 快进不是稳定 8x。截图里 Run A 底部显示 actual x7.4，Run B 降到 actual x4.0。对普通玩家这只是性能问题；对速通玩家这是可重复性问题。如果同样 30 秒真实时间，模拟步数不同，benchmark 和路线比较就不公平。

AI 的寻食优先级看起来不够硬。工人显示 well-fed、hungry、starving 混杂，同时仍有人在 Process、Deliver、Seek Task。若一个工人马上饿死，AI 应该更激进地中断任务去找食物，或者 UI 至少解释为什么不能吃。

Autopilot 会推进生产，但不会保证生存底线。它能做 lumber route、smith/process 等发展动作，却可能放任饥饿死亡。作为 fallback，它还可以；作为默认推荐，它会误导玩家。

资源短缺反馈不足。Run B 暂停铺设后出现 `Insufficient resources`，但大量点击里哪些成功、哪些失败、哪些排队、哪些无效并不清楚。优化玩家需要的是可审计队列，而不是只看最终尸体。

人口显示很重要但惩罚不够突出。Run B 人口掉到 8，仍显示同样 Survived/Score/Dev，这会让玩家怀疑人口是否只是消耗品。如果人口是核心价值，应让 Score/Dev/事件强烈反映损失。

## 游戏是否有真正玩法上限

目前有“局部玩法上限”，但还没有“真正玩法上限”。

局部上限来自道路布局、资源节点连接、厨房/农田比例、仓库位置、工人职业分配和快进节奏。玩家确实可以优化这些东西，尤其是路网和 depot 连接。接管局 Dev 60、人口 32、食物 200+，说明系统可以进入稳定繁荣状态。

但真正上限需要三件事：可重复的开局、明确的失败归因、多个互相制衡的路线。当前这三项都偏弱。开局重复后 Survived/Score/Dev 可以相似，但人口结果差异极大；失败归因常被 hunger 黑箱覆盖；路线之间看起来不像“食物爆发 vs 工业速推 vs 防御龟缩”，更像“路修得够不够、AI 有没有饿死人”。

如果想让速通玩家长期研究，游戏需要更明确的目标分层。例如 5 分钟内 Dev 80、零死亡、修复 east depot、抵御第一波威胁、维持 meals surplus 等。否则玩家只会找到一条免费道路 + depot + AI fallback 的路线，然后缺少继续压榨的理由。

## 改进建议

1. 给 Road 增加真实机会成本。可以是木材维护、施工时间更长、道路上限、连接距离税、或者道路必须由 warehouse/flag 分段供给。免费但强力的路网很容易成为唯一答案。

2. 强化食物闭环的可解释性。在工人死亡或 starvation 前显示“为什么没吃到食物”：无可达食物、厨房未完成、农田未产出、物流堵塞、AI 被任务锁定等。

3. 把暂停建造变成规划模式而不是免费超能力。暂停时可以预览，但恢复时才确认资源锁定；或者暂停下单有队列上限；或者高难度/计分模式禁用暂停建造。

4. 让 Autopilot 有策略档位。至少提供 Food-first、Depot-rush、Balanced、Defense 四种模式，并显示它下一步为什么这样做。否则 Autopilot 既强又不可控。

5. 让 Score 更重视人口质量。死亡、饥饿、长时间 hungry 应显著扣分，避免“饿死一半人但 Dev 还不错”的路线变成合理选择。

6. 提供 run-end 或 benchmark 面板。显示 Survived、Score、Dev、Deaths、Births、Meals produced、Idle time、Haul distance、Road tiles built、Buildings completed、Threat damage。没有这些，玩家很难比较策略。

7. 稳定 8x 的模拟步数。至少在 UI 上区分“目标倍速”和“实际模拟 ticks”，并允许固定 tick benchmark。速通/优化评测需要确定性。

8. 增强建筑放置反馈。暂停批量点建筑时，每个格子应有明确状态：可建、缺资源、不可达、等待施工、规则不满足、会阻断路径。

9. 给仓库和墙体明确定位。仓库应该改变物流数学，墙体应该改变威胁路径；如果只是“能放但收益不清楚”，优化玩家不会把它们纳入路线。

10. 做一个“零死亡奖惩”。这会立刻把 food-first、clinic、路径优化和 AI 救援变成真实目标，而不是只追 Dev。

## 一句话总结

当前最像最优解的是“免费道路优先 + depot 连接 + Autopilot/8x 执行”，但食物死亡、暂停规划和快进不稳定让策略比较变得黑箱，游戏还需要更强的数值可解释性和反 cheese 约束。
